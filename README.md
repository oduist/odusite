Odusite is a radical architectural shift for Odoo. It re-creates the entire
public website and customer portal of **Odoo 19** as a decoupled, high-performance
[Astro](https://astro.build) frontend on **Cloudflare Workers**, with Odoo demoted
to a pure, invisible backend.

Odoo's own rendering (QWeb, snippets, website builder) is **not** used. The Astro
site talks to Odoo exclusively through a REST API (`/odusite/v1/...`), authenticated
server-to-server with the `X-Odusite-Token` header. It targets a new era where web
interfaces are generated and maintained by AI agents — not hand-assembled in a
visual CMS — and where exposing an ERP to the public internet is an unacceptable
risk.

```
addons/      Odoo 19 addons (odusite_base, odusite_sale, odusite_portal, ...)
site/        Astro site: build-time blocks, themes, Cloudflare Worker
specs/       Specifications — the system is re-creatable from them
docs/admin/  Administrator documentation
docs/user/   End-user documentation
```

---

## 🛑 The monolith trap: why the classic Odoo CMS is obsolete

Historically, Odoo bundles the ERP, the database, and the public website into a
single monolithic block. Convenient a decade ago, this is fundamentally flawed for
modern needs:

- **Shared attack surface.** Public web traffic hits the exact server that holds
  your accounting, inventory, and corporate secrets. A single frontend-template
  vulnerability or a DDoS spike can expose the database or paralyze internal
  operations.
- **The "human-only" CMS bottleneck.** Odoo's WYSIWYG builder is drag-and-drop by
  design. You cannot programmatically control a visual CMS — but UI, content, and
  localization are increasingly generated by **AI agents**.
- **Performance degradation.** Rendering heavy QWeb templates for every visitor
  drains the same workers you need for ERP calculations, hurting page loads and
  Core Web Vitals.

---

## 💡 The decoupled paradigm

Odusite physically and logically separates the public face from the business brain.

- **Odoo off the public attack surface.** The ERP and the website no longer breathe
  the same air. Odoo runs in a private network with no public ports; the browser
  never talks to it. The recommended topology reaches Odoo only through an
  authenticated **Cloudflare Tunnel** (`cloudflared`), so all traffic is
  Worker → Odoo, server-to-server — enforced by ADR-002 (`X-Odusite-Token`,
  constant-time compared). See `specs/decisions.md`.
- **AI-native workflow.** Marketing content lives as flat files in the repo (Astro
  content collections, MDX, components) — the perfect substrate for agents.
  Containerized agents can generate content and components and commit directly;
  CI/CD builds and deploys with no click in an admin panel.
- **Edge performance.** Astro edge SSR on Cloudflare Workers serves pages fast:
  static marketing pages are prerendered and edge-cached, while catalog/cart/portal
  are SSR with tag-based cache invalidation pushed by Odoo webhooks. Public traffic,
  caching, and media (via **Cloudflare R2**, zero egress) are offloaded to the CDN,
  leaving Odoo free for business logic.
- **Stateless scalability.** No Odoo session cookies. The portal authenticates with
  short-lived JWTs + rotating refresh tokens; the cart is a draft `sale.order`
  addressed by id + access token in a first-party cookie. Traffic spikes scale on
  the edge and never crash the ERP.

## 🛠️ Technology stack

- **Frontend & routing:** Astro 5, TypeScript strict — edge SSR, View Transitions,
  build-time **blocks** (blog, shop, portal, events, jobs, ...) and token-based
  **themes**.
- **Edge infrastructure:** Cloudflare Workers (`@astrojs/cloudflare`) + Cache API.
- **Storage (zero egress):** Cloudflare R2 / any S3-compatible store via
  `odusite_s3` for images, PDFs, and attachments.
- **Backend & transactions:** Odoo 19 (private network) with lightweight custom JSON
  controllers in the `odusite_*` addons — no external pip dependencies.
- **Security:** strict `X-Odusite-Token` server-to-server auth, portal JWT (HS256),
  Cloudflare WAF/Turnstile on public forms, and a Cloudflare Tunnel to keep Odoo
  private.
- **Voice navigation (optional):** an ElevenLabs Conversational AI agent that drives
  the site hands-free; the API key stays server-side and it ships zero JS when off.

## 🎙️ Voice navigation (optional)

Fitting the AI-native theme, Odusite ships an optional hands-free voice assistant,
embedded site-wide and **off until configured**. A visitor taps the mic and talks to
an [ElevenLabs](https://elevenlabs.io) Conversational AI agent that answers questions
and **navigates the site for them**: the agent calls `navigate` / `search_site` client
tools wired to Astro's View Transitions router, so it keeps talking while pages change.
The ElevenLabs API key never reaches the browser — the Worker mints short-lived signed
WebSocket URLs — and when `ELEVENLABS_AGENT_ID` is unset the widget contributes no JS
at all. It's a pure site feature with no Odoo involvement. Setup:
`docs/admin/voice-assistant.md`; design: `specs/site/04-voice-assistant.md`.

## 🚀 Quick start

**Odoo**: add `addons/` to the addons path, install the `odusite_*` modules you need
(`odusite_base` is mandatory), copy the API token from Website Settings → Odusite.
See `docs/admin/installation.md`.

**Site**:

```bash
cd site
pnpm install
echo 'ODUSITE_TOKEN=<token from Odoo>' >> .dev.vars
echo 'ODOO_URL=http://localhost:8069' >> .dev.vars
pnpm dev
```

Deploy: `docs/admin/site-deploy.md`. Architecture: `specs/01-architecture.md`.
Decisions (ADRs): `specs/decisions.md`. Contributor guide: `CLAUDE.md`.

## 📦 Modules

Each addon extends one stock Odoo module and exposes only its REST API — everything
shared (token gate, JSON envelope, pagination, image URLs, webhooks) lives in
`odusite_base`. A module's Astro counterpart is a build-time **block** in `site/`.

| Addon | Builds on | What it exposes |
|---|---|---|
| `odusite_base` | `website` | API core: `X-Odusite-Token` gate, `odusite_route()` helper, JWT utils, JSON envelope/pagination, cache-invalidation webhook queue, site/menus/sitemap/redirects/search endpoints. **Mandatory.** |
| `odusite_portal` | `portal`, `auth_signup` | Headless auth: JWT login/refresh/logout, self-service signup with email double opt-in, password reset, `/me` profile & addresses, generic chatter. No Odoo sessions. |
| `odusite_sale` | `website_sale` | eCommerce API: catalog & facets, product detail, stateless cart (draft `sale.order` by id + access token), checkout state machine, portal orders (sign/decline/PDF). |
| `odusite_payment` | `payment` | Headless payments: compatible providers/methods/saved tokens, create `payment.transaction` (Stripe PaymentIntent client secret), transaction polling. PSP webhooks stay on stock routes. |
| `odusite_account` | `account` | Portal invoices: paginated posted invoices, detail with lines/totals, legal PDF stream; payable via `odusite_payment`. |
| `odusite_blog` | `website_blog` | Blogs, posts and tags read API, plus sitemap and cache-invalidation hooks. |
| `odusite_crm` | `website_crm` | Website forms → `crm.lead` (UTM mapping, sales team), whitelisted generic form endpoint, honeypot + per-IP throttle. |
| `odusite_event` | `website_event` | Events listing/detail with tickets & seats, free-ticket registration, ICS export, sitemap/webhooks. |
| `odusite_slides` | `website_slides` | eLearning: courses listing/detail, join, slide content & downloads, completion tracking, quizzes. |
| `odusite_forum` | `website_forum` | Q&A forum: public reads + JWT actions (ask/answer/vote/accept/comment), karma enforced and mapped to API errors. |
| `odusite_hr_recruitment` | `website_hr_recruitment` | Jobs listing/detail with facets and multipart application (CV upload). |
| `odusite_project` | `project` | Portal projects & tasks: lists/detail, attachments, subtasks, timesheet summary, task suggestion form. |
| `odusite_partner` | `website_partner` | Public partner directory with country/grade/tag facets and detail pages. |
| `odusite_mass_mailing` | `website_mass_mailing` | Newsletter subscription → `mailing.contact`/`mailing.subscription`, public mailing lists, honeypot. |
| `odusite_s3` | `odusite_base` | Offloads the `ir.attachment` filestore to S3/Cloudflare R2 (zero egress): `s3://` routing, reference-counted GC, throttled background migration, presigned direct download, hybrid public delivery. |

Full contracts (routes, models, fields, flows) live in `specs/modules/`.

## 📜 License & contributions

Odusite is open source under the **MIT License** — Copyright © 2026 Oduist OÜ. See
`LICENSE`; every `odusite_*` addon declares `'license': 'MIT'`.

Odusite is **not** meant to be a one-size-fits-all, turnkey product. Think of it as
a starting point — a set of building blocks and semi-finished parts (API controllers,
blocks, themes) that you fork and finish for your own needs. The intended workflow is
an AI agent growing the app individually around your requirements, so what ships here
is scaffolding to build on, not a website to install and forget.

Contributions are welcome but entirely optional. If you build something genuinely
universal and minimal enough to belong upstream, open a pull request — but that is the
exception, not the expectation. See `CLAUDE.md` for conventions and the definition of
done.
